# Internationalisation

HEXA Structures utilise l'internationalisation Qt côté GUI uniquement :

- `QTranslator` charge les fichiers compilés `.qm`.
- Les sources de traduction sont les fichiers `.ts` dans `i18n/`.
- Le français reste la langue source par défaut.
- Le core métier ne dépend pas de PySide6 pour les traductions.
- Le choix de langue est sauvegardé avec `QSettings`.

Pour l'instant, seules les langues prêtes sont visibles dans le menu de l'application (`fr`, `en`). Les fichiers `hexa_es.*` et `hexa_de.*` restent présents comme squelettes mais ne sont pas exposés tant que les traductions ne sont pas suffisamment complètes.

## État actuel

Les langues visibles dans `Paramètres > Langue` sont générées depuis `LanguageManager.available_languages` :

```python
available_languages = {
    "fr": "Français",
    "en": "English",
}
```

Les fichiers suivants existent déjà pour préparer l'extension future :

```text
i18n/hexa_fr.ts
i18n/hexa_en.ts
i18n/hexa_es.ts
i18n/hexa_de.ts

i18n/hexa_fr.qm
i18n/hexa_en.qm
i18n/hexa_es.qm
i18n/hexa_de.qm
```

Les langues `es` et `de` sont volontairement masquées dans le menu tant que leurs catalogues ne contiennent pas de traduction exploitable.

Le balayage actuel couvre notamment :

- les menus et actions principales de `MainWindow` ;
- les panneaux principaux : arbre du modèle, propriétés, résultats, tables et diagrammes ;
- les boîtes de dialogue les plus visibles : grille, matériaux, sections, sections shell, charges, combinaisons, paramètres Eurocodes, conditions aux limites, propriétés d'éléments, maillage de plaques et export ;
- les listes déroulantes utilisateur les plus visibles : types d'appuis, types de charges, catégories Eurocode, familles de résultats, choix de diagrammes et options de sections ;
- les ajouts récents de sections : aperçu dynamique, sections paramétriques I/U/L/tubes, limites géométriques et repère/orientation locale des barres.
- le Section Builder : contours, trous, import de profils, maillage, contraintes
  et note de calcul.

La traduction n'est pas considérée comme terminée sur tout le logiciel. Les nouveaux textes visibles doivent continuer à passer par `self.tr(...)` ou par un mapping GUI traduisible.

Au 19 juin 2026, le test automatisé confirme que le catalogue anglais ne
contient aucune entrée vide ou marquée `unfinished`.

## Ajouter une langue

1. Créer ou régénérer le catalogue source `.ts`.

La commande demandée par le projet reste :

```powershell
pyside6-lupdate gui -ts i18n/hexa_en.ts
```

Sur ce projet, passer explicitement les fichiers `.py` et `.ui` à `lupdate` pour ramasser les widgets Python et les formulaires Qt Designer :

```powershell
$files = @(Get-ChildItem -Path gui -Recurse -Include *.py,*.ui | ForEach-Object { $_.FullName })
pyside6-lupdate @files -ts i18n/hexa_en.ts
```

Pour une autre langue :

```powershell
$files = @(Get-ChildItem -Path gui -Recurse -Include *.py,*.ui | ForEach-Object { $_.FullName })
pyside6-lupdate @files -ts i18n/hexa_es.ts
```

La forme courte peut ne pas scanner récursivement tous les fichiers Python selon la version de PySide. En cas de doute, préférer la commande PowerShell avec la liste explicite de fichiers.

2. Ouvrir le fichier `.ts` dans Qt Linguist ou un éditeur compatible, puis renseigner les traductions.

3. Compiler le fichier `.qm` :

```powershell
pyside6-lrelease i18n/hexa_en.ts -qm i18n/hexa_en.qm
```

Pour une autre langue :

```powershell
pyside6-lrelease i18n/hexa_es.ts -qm i18n/hexa_es.qm
```

4. Ajouter la langue dans `LanguageManager.available_languages` uniquement quand elle doit être visible dans l'interface :

```python
available_languages = {
    "fr": "Français",
    "en": "English",
    "es": "Español",
    "de": "Deutsch",
    "it": "Italiano",
}
```

Le menu `Paramètres > Langue` est construit depuis ce dictionnaire. Ne pas ajouter de condition spécifique du type `if language == "en"` pour charger les catalogues.

5. Tester le chargement de la langue depuis l'application ou depuis les tests automatisés.

## Régénérer les catalogues existants

Après avoir ajouté ou renommé des appels `self.tr(...)`, régénérer le catalogue anglais :

```powershell
pyside6-lupdate gui -ts i18n/hexa_en.ts
pyside6-lrelease i18n/hexa_en.ts -qm i18n/hexa_en.qm
```

Puis régénérer le catalogue français si nécessaire :

```powershell
pyside6-lupdate gui -ts i18n/hexa_fr.ts
pyside6-lrelease i18n/hexa_fr.ts -qm i18n/hexa_fr.qm
```

Si `lupdate gui` ne ramasse pas tous les fichiers dans l'environnement local, utiliser la variante PowerShell avec `$files`.

Après une extraction, vérifier immédiatement que le catalogue anglais ne contient plus
d'entrée non traduite :

```powershell
Select-String -Path i18n\hexa_en.ts -Pattern 'type="unfinished"'
```

Le test `tests/test_i18n.py` échoue volontairement si `hexa_en.ts` contient encore
`type="unfinished"` ou une traduction vide. Les nouvelles chaînes très visibles doivent
donc être traduites avant de recompiler `hexa_en.qm`.

## Convention de nommage

Les fichiers suivent toujours cette convention :

```text
i18n/hexa_<code>.ts
i18n/hexa_<code>.qm
```

Exemples :

```text
hexa_fr.ts
hexa_en.ts
hexa_es.ts
hexa_de.ts
```

Le chargeur utilise le code langue pour construire le chemin :

```python
qm_path = i18n_dir / f"hexa_{language_code}.qm"
```

Si le `.qm` est absent ou invalide, `LanguageManager` ne doit pas faire planter l'application. Le français reste le repli par défaut.

## Retraduction de l'interface

`MainWindow.retranslate_ui()` remet à jour les textes principaux après un changement de langue :

- titre de fenêtre ;
- menus ;
- actions ;
- boutons principaux ;
- panneaux et widgets principaux.

Les widgets importants peuvent exposer leur propre `retranslate_ui()` quand ils gardent des textes en état interne. Après un changement de langue, la fenêtre principale appelle ces méthodes quand elles existent.

Pour les libellés provenant de valeurs métier ou de listes internes, utiliser un mapping côté GUI, par exemple dans `gui/i18n/display_labels.py`. Le core doit continuer à retourner des codes ou des valeurs stables, sans importer PySide6.

## Packaging

Le spec PyInstaller inclut déjà le dossier :

```python
("i18n", "i18n")
```

Pour un build Nuitka, ajouter l'option équivalente :

```powershell
--include-data-dir=i18n=i18n
```

Vérifier que les fichiers `.qm` sont présents dans le dossier `i18n/` du build final. Sans ces fichiers compilés, l'application revient proprement au français source.

## Tests utiles

Tests rapides liés à l'i18n :

```powershell
pytest tests\test_i18n.py
```

Tests ciblés après modification de textes dans les dialogues de sections :

```powershell
pytest tests\test_i18n.py tests\test_surface_sections_ui.py
```

Tests ciblés après modification d'autres dialogues visibles :

```powershell
pytest tests\test_i18n.py tests\test_boundary.py tests\test_plate_dialogs.py
```

Quand la modification touche `MainWindow`, relancer aussi les tests de toolbar et de supports de surface :

```powershell
pytest tests\test_main_window_toolbar.py tests\test_main_window_surface_support.py
```

## Bonnes pratiques

- Utiliser `self.tr("Texte visible")` dans les widgets PySide6.
- Ne pas traduire le core métier directement.
- Préférer des codes métier côté core, puis traduire dans la GUI plus tard, par exemple `plate_requires_four_nodes`, `section_missing`, `invalid_material`.
- Ajouter les textes progressivement : menus, actions, panneaux principaux, puis dialogues et messages.
- Garder les textes source en français quand ils correspondent à l'interface historique.
- Regrouper les libellés métier traduisibles côté GUI plutôt que dans les modèles de calcul.
- Regénérer les `.qm` après chaque modification significative des `.ts`.
